Crypto Necessity Check

If blockchain is removed, what breaks?

  • Credible, tamper-resistant settlement history shared across prediction apps.
  • Permissionless disputing with bonded economic incentives.
  • Composable final outcome that markets can settle against without trusting one operator.

Result: Onchain architecture is justified, but only if you can prove at least one market actually needs this over centralized moderation.

Top Gaps To Fix

1) No hard demand evidence yet (High)

Docs show strong mechanism design but no signed LOI, no pilot partner, and no user interviews from market operators.

2) Scope too broad for hackathon timeline (High)

Switchboard + two-stage disputes + MagicBlock private voting + dual-token economics is too much for one cycle if quality is required.

3) Token bootstrap ambiguity (High)

PUSD and OPAL are central, but issuance, liquidity, and participant onboarding are unspecified. This can block real usage in demo conditions.

4) Liveness and latency SLAs missing (Medium)

No explicit target for dispute-to-finalization time. Integrators need predictable settlement windows.

5) Governance capture risk not sufficiently bounded (Medium)

OPAL-weighted voting can be captured without anti-whale or quorum-hardening details.

6) Fallback path under infra failure missing (Medium)

No explicit behavior if Switchboard quote is unavailable, stale, or malformed during critical windows.

Hackathon Improvement Plan (Practical)

  1. Cut to an MVP lane: implement assertion + first dispute + Switchboard LLM resolution + finalization; defer full OPAL voting to mocked or admin-gated fallback in demo.
  2. Land one design partner: secure one prediction market, game, or social betting app to integrate your resolution API in hackathon timeframe.
  3. Publish resolver SLA: set max times for liveness, LLM challenge, and finalization (for example, 5m/10m/15m in demo settings).
  4. Define token simplification for hackathon: use one stable collateral token for demo, and treat OPAL governance as post-hackathon roadmap.
  5. Produce adversarial demo cases: one clean true case, one false disputed case, one too-early case, one unresolvable case.
  6. Ship integrator SDK + tiny UI: judges should settle a toy market in under 60 seconds using your SDK docs.
  7. Add trust report: per assertion, show evidence hash, quote slot freshness, and why outcome was accepted.
Final recommendation: Build this for Frontier only as a focused integration-first MVP. Do not attempt full protocol completeness during the event. Win by proving one painful user workflow is solved end-to-end with credible incentives and clear finality.
