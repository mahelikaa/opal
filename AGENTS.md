# CLAUDE.md

Behavioral guidelines to reduce common LLM coding mistakes. Merge with project-specific instructions as needed.

**Tradeoff:** These guidelines bias toward caution over speed. For trivial tasks, use judgment.

## 1. Think Before Coding

**Don't assume. Don't hide confusion. Surface tradeoffs.**

Before implementing:

- State your assumptions explicitly. If uncertain, ask.
- If multiple interpretations exist, present them - don't pick silently.
- If a simpler approach exists, say so. Push back when warranted.
- If something is unclear, stop. Name what's confusing. Ask.

## 2. Simplicity First

**Minimum code that solves the problem. Nothing speculative.**

- No features beyond what was asked.
- No abstractions for single-use code.
- No "flexibility" or "configurability" that wasn't requested.
- No error handling for impossible scenarios.
- If you write 200 lines and it could be 50, rewrite it.

Ask yourself: "Would a senior engineer say this is overcomplicated?" If yes, simplify.

## 3. Surgical Changes

**Touch only what you must. Clean up only your own mess.**

When editing existing code:

- Don't "improve" adjacent code, comments, or formatting.
- Don't refactor things that aren't broken.
- Match existing style, even if you'd do it differently.
- If you notice unrelated dead code, mention it - don't delete it.

When your changes create orphans:

- Remove imports/variables/functions that YOUR changes made unused.
- Don't remove pre-existing dead code unless asked.

The test: Every changed line should trace directly to the user's request.

## 4. Goal-Driven Execution

**Define success criteria. Loop until verified.**

Transform tasks into verifiable goals:

- "Add validation" → "Write tests for invalid inputs, then make them pass"
- "Fix the bug" → "Write a test that reproduces it, then make it pass"
- "Refactor X" → "Ensure tests pass before and after"

For multi-step tasks, state a brief plan:

```
1. [Step] → verify: [check]
2. [Step] → verify: [check]
3. [Step] → verify: [check]
```

Strong success criteria let you loop independently. Weak criteria ("make it work") require constant clarification.

---

**These guidelines are working if:** fewer unnecessary changes in diffs, fewer rewrites due to overcomplication, and clarifying questions come before implementation rather than after mistakes.

# Opal Project Reference

## Build & Test Commands

```bash
# Full build + codegen + test (localnet)
bun run setup && bun run test

# Build only (Rust program + IDL)
anchor build

# Codegen only (IDL → JS client)
bun run generate       # codama run js

# Rust unit tests (uses LiteSVM, no localnet required)
cargo test -p opal

# TypeScript tests (requires localnet validator running)
bun run test           # anchor test

# Devnet tests (pre-deployed program, no build/deploy)
bun run test:devnet

# Format check
bun run format         # prettier tests/* --check
```

**Build order matters:** `anchor build` → `bun run generate` (codama reads `target/idl/opal.json`). The TS test imports `@client/index` which resolves to the generated Codama client in `web/client/src/generated/`. If tests fail on import errors, run `bun run generate` first.

## Architecture

Single Anchor program at `programs/opal/` (Anchor 0.32.1, Rust toolchain pinned to 1.89.0 in `rust-toolchain.toml`). Package manager is **Bun** (not npm/yarn). TS tests use `bun:test` + `@solana/kit` + `solana-kite`.

Program ID (localnet): `8NCcxyAzKiAHxJ9DMnADtxShYutS9w81wHcXqgCavTBy`

**Source layout:**

- `programs/opal/src/instructions/` — one file per instruction, each exporting a `handler(ctx, args) -> Result<()>` free function
- `programs/opal/src/state/` — Anchor account structs with `#[account]` and `#[derive(InitSpace)]`
- `programs/opal/src/constants.rs` — PDA seeds (`b"protocol_config"`, `b"assertion"`, etc.) and max lengths
- `programs/opal/src/errors.rs` — `OpalError` enum (do NOT use `TemplateError`)
- `web/client/` — **gitignored**, generated by Codama from the Anchor IDL
- `tests/opal.test.ts` — integration tests using generated Codama instruction getters

**State machine:** `Asserted` → (disputed?) `PendingLlm` → `AssertedLlm` → (challenged?) `PendingVote` → `Voting` → `Resolved`. Undisputed path: `Asserted` → `Resolved` via `finalize_undisputed`.

## Gotchas

- **Dead template files:** `instructions/initialize.rs`, `instructions/increment.rs`, `instructions/decrement.rs` are Anchor scaffold leftovers. They reference `TemplateError` and `Counter` which don't exist. They're NOT wired into the program's `#[program]` block in `lib.rs`. Leave them alone; do not try to fix their imports.

- **PDA derivation for assertions uses a caller-supplied `assertion_id`**, not auto-derived from other fields. See `create_assertion.rs` — the seed is `[ASSERTION_SEED, args.assertion_id.as_ref()]`.

- **Switchboard and MagicBlock fields are reserved but not wired.** The `LlmResolutionRound` stores Switchboard fields that default to `Pubkey::default()` / zeros. The `VoteResolutionRound` has MagicBlock fields but no delegation logic (`delegated` flag is set true in `open_vote` but no ER delegation is performed). This is intentional for the current stage.

- **Anchor.toml** sets `package_manager = "bun"` and test script = `cargo test -p opal`. The Rust test runner in `programs/opal/tests/` uses LiteSVM.

- **No CI yet** — no `.github/` directory exists.

- **`.env`** may be needed for devnet tests (`KEYPAIR_BYTES`, `CLUSTER`).
