# Single trusted off-chain LLM resolver for the MVP; the Switchboard council is dropped

The first-pass dispute resolution is a **single LLM call**, posted on-chain by a trusted, authority-gated off-chain resolver (productionizing the existing mock path). It records the verdict and timing on-chain. Binding LLM provenance hashes on-chain for auditability (the currently-unused `*_hash` fields) is **not** part of this MVP — it is deferred to the `Vision` trust-minimized/permissionless resolver. This supersedes the three-feed Switchboard On-Demand "council", which remains the current non-mock resolution path in code (`submit_llm_resolution`) until it is removed with this change (dropping the `switchboard-on-demand` dependency).

**Why.** The LLM layer does **not** need to be trustless, because the staked vote is the trust backstop ([ADR-0003](0003-private-staked-voting.md)): a wrong or corrupt verdict is challengeable into the trust-minimized vote. The council was buying trust-minimization at a layer that is _already_ backstopped, at the cost of operating three live LLM feeds. The council resolution instruction (`submit_llm_resolution`) is implemented and compiled, but was never operationally stood up — no live feeds were ever configured and tests only ever ran the mock — so collapsing it loses no working functionality. A trusted resolver ships fastest and is fully under our control while we also absorb the MagicBlock integration.

**Considered options.** 3-feed council (rejected: operational overhead, unbuilt, redundant with the vote); single Switchboard feed (rejected for MVP: still net-new feed/job work plus a live external dependency); **trusted resolver (chosen)**.

**Consequences.** The MVP first pass is centralized and must be trusted to call the LLM honestly — acceptable _only_ because the vote can override it. Trust-minimized inference (Switchboard, TEE-attested, or otherwise permissionless) is the documented `Vision` upgrade, and any on-chain LLM provenance hashing would land there. The Switchboard-specific fields on `LlmResolutionRound` (and the currently-unused `*_hash` fields) are unused and may be removed; the account shape for any future resolver — trusted or permissionless — is not settled here.
